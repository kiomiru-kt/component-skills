# figma-extract

FigmaのComponentsページを走査し、エンジニアが命名したフレームをもとにコンポーネントセット・バリアントプロパティをFigmaに構築するスキル。

## 前提条件

- Figma MCPが接続済みであること
- エンジニアがFigmaの `Components` ページにコンポーネント候補を配置・命名済みであること
- フレーム名は `figma-properties.md` のプロパティ記法（`{キー}={値}[,{キー}={値}...]`）で命名されていること

## トリガー時に受け取る情報

- 走査対象のFigmaファイルURL または file-id
- （任意）対象コンポーネント名の絞り込み

## 実行フロー

### Step 1: Figma MCP 接続確認

Figma MCPが利用可能か確認する。

**接続されていない場合 → 即座に停止:**

```
Figma MCPが接続されていません。
接続後に再度実行してください。
```

### Step 2: `figma-properties.md` を読み込む

`~/.claude/figma-properties.md` を読み込み、使用可能なプロパティキーと値の一覧を把握する。

ファイルが存在しない場合:

```
~/.claude/figma-properties.md が見つかりません。
グローバルドキュメントが正しくインストールされているか確認してください。
```

### Step 3: `Components` ページを走査

`get_metadata` でFigmaファイルのページ一覧を取得し、`Components` ページのIDを特定する。

**`Components` ページが存在しない場合 → 停止してエンジニアに確認:**

```
Figmaに "Components" ページが見つかりません。
ページ名を確認してください（大文字・小文字を区別します）。
```

`get_design_context` で `Components` ページ内の全フレームを取得する。
- 取得対象: フレーム名、子要素構造
- 走査範囲: `Components` ページのみ（他ページは対象外）

### Step 4: フレームをパースしてグループ化

各フレーム名を以下のルールでパースする:

```
{コンポーネント名} / {プロパティ文字列}
例: heading-01 / device=sp,align=left,color=main
```

- スラッシュ区切りでコンポーネント名とプロパティを分離
- プロパティ文字列をカンマ区切りで `{キー}={値}` に分解
- 同一コンポーネント名のフレームをグループ化

**プロパティ記法に合致しないフレームが存在する場合:**

```
以下のフレームはプロパティ記法に合致しないため除外しました:
- {フレーム名}: {理由}

除外せず処理する場合は指示してください。
```

**未知のプロパティキーが登場した場合（`figma-properties.md` に存在しない）:**

```
未知のプロパティキー "{key}" が含まれています（フレーム: {フレーム名}）。
処理を続けますか？ それとも figma-properties.md を更新しますか？
```

→ **エンジニアの判断を待ってから続行する。推測で進めない。**

### Step 5: 構造の一貫性チェック

同一コンポーネント名でフレームの子要素構造が大きく異なる場合:

```
"{コンポーネント名}" に構造が異なるフレームが含まれています:
- {フレームA}: {構造の概要}
- {フレームB}: {構造の概要}

同一コンポーネントとして統合しますか？
それとも別コンポーネント（例: {name}-01 / {name}-02）として分割しますか？
```

→ **エンジニアの確認を必ず取る。推測で統合・分割しない。**

### Step 6: Figmaにバリアントプロパティを設定

グループ化した各コンポーネントに対し `use_figma` でバリアントプロパティを設定する:

- プロパティ名: キー名をそのまま使用（`device` / `state` / `align` / `color` / `size`）
- プロパティ値: 各バリアントフレームの値を設定

**書き込み前にエンジニアへ確認を取る（破壊的操作のため）:**

```
以下のコンポーネントセットをFigmaに構築します:

| コンポーネント名 | バリアント数 | プロパティ |
|----------------|------------|---------|
| heading-01     | 4          | device / align / color |
| btn-01         | 3          | state |

この内容でFigmaに書き込んでよいですか？
```

→ 承認後のみ `use_figma` を実行する。

### Step 7: 整理レポートを出力

`docs/figma-audit/components-{YYYY-MM-DD}.md` に以下の形式で出力する:

```markdown
## figma-extract 整理レポート {YYYY-MM-DD}

### 構築したコンポーネントセット
| コンポーネント名 | BEMクラス | バリアント数 | プロパティ |
|----------------|---------|------------|---------|
| heading-01 | .c-heading-01 | 4 | device / align / color |

### 要確認事項（エンジニア判断が必要）
- {コンポーネント名}: {理由}

### 未処理フレーム
- {フレーム名}: {除外理由}
```

`docs/figma-audit/` ディレクトリが存在しない場合は作成する。

### Step 8: figma-tokenize を連続実行

整理レポート出力後、figma-tokenize スキルを連続実行する。

## エッジケース一覧

| 状況 | 対応 |
|------|------|
| Figma MCP 未接続 | Step 1 で即停止・案内 |
| `Components` ページが存在しない | Step 3 で停止・ページ名確認を促す |
| `figma-properties.md` が存在しない | Step 2 で停止・インストール案内 |
| プロパティ記法外のフレーム名 | 除外リストに追加・エンジニアに通知 |
| 未知のプロパティキー | Step 4 で停止・エンジニアに確認 |
| 同一名で構造が大きく異なるフレーム | Step 5 で停止・統合/分割を確認 |
| `use_figma` 書き込みエラー | エラー内容を表示して停止 |

---

## トークン最適化

- 走査対象は `Components` ページのみ（全ページ走査禁止）
- プロパティ規則は `~/.claude/figma-properties.md` を参照する（SKILL.md にハードコードしない）
- バリアブル抽出（数値処理）は figma-tokenize の Python スクリプトに委譲する
