# spec-builder

ページ・セクション・コンポーネント・機能単位の実装仕様書（spec.md）・実装指示書（implementation.md）・完了チェックリスト（checklist.md）を生成するスキル。

Figma URLがあれば自動取得（Mode A）、なければ対話形式（Mode B）、機能キーワードがあればリファレンス参照（Mode C）で動作する。

---

## 実行フロー

### Step 1: モード判定

入力を解析し、以下の優先順位でモードを決定する:

| 条件 | モード |
|------|--------|
| Figma URLが含まれている | **Mode A**（Figma自動取得） |
| 検索・スライダー等の機能キーワードが含まれている | **Mode C**（機能リファレンス参照） |
| どちらも該当しない | **Mode B**（対話形式） |

Mode A でも不足情報がある場合は Mode B の質問で補完する。

### Step 2（Mode A）: Figmaデータを取得

Figma MCPが利用可能か確認する。

**Figma MCP 未接続の場合 → Mode B へフォールバック:**

```
Figma MCPが接続されていないため、対話形式（Mode B）で情報収集します。
```

接続済みの場合:

1. `get_metadata` でノード構造を取得
2. `get_design_context` で対象フレームの詳細を取得
3. `get_screenshot` でビジュアルを確認
4. 取得できなかった情報のみ Mode B の質問で補完する

取得した数値は仕様書にそのまま記載する。**AIによる推測・補完は禁止。取得できない場合は「未定」として記録する。**

### Step 2（Mode B）: 対話形式で情報収集

以下の質問を **1問ずつ** 順番に行う。不明な項目は「未定」として記録して次へ進む。

**共通質問（ページ・コンポーネント両方）:**

1. 何を作るか（1文で）
2. レスポンシブ仕様（SP / PC のデザイン差分、ブレークポイント）
3. インタラクション（hover / active / アニメーション有無）

**ページ・セクション固有:**

4. WordPressテンプレート種別（archive / single / page / front-page / template-parts）
5. データ取得（投稿タイプ・ACF/SCFカスタムフィールド・WP_Queryパラメータ・ページネーション有無）
6. SEO（メタ情報・構造化データの有無）

**コンポーネント固有:**

4. BEMクラス名（次のステップで既存確認後に確定）
5. バリアント（状態一覧）
6. 引数（PHPテンプレートに渡す `$args` の項目）

### Step 2（Mode C）: 機能リファレンスを参照

機能キーワードを検出し、対応するリファレンスドキュメントを参照する:

| キーワード | 参照ドキュメント |
|-----------|----------------|
| 検索、search、キーワード | `docs/features/search.md` |
| ページネーション、pagination | `docs/features/pagination.md` |
| ループ、WP_Query、投稿一覧、archive | `docs/features/post-loop.md` |
| ナビゲーション、nav、ハンバーガー | `docs/features/navigation.md` |
| タブ、tab、切り替え | `docs/features/tabs.md` |
| スライダー、slider、カルーセル、swiper | `docs/features/slider.md` |
| カスタムブロック、Lazy Blocks、Gutenberg | `docs/features/custom-blocks.md` |

**参照後、リファレンスドキュメントの充実度を評価する:**

- ドキュメントが存在しない、または「実装例なし」「TODO」のみの場合 → **空ドキュメントとして扱い、後述の Step 5 でコンテンツ提案を行う**
- 実装例が記載されている場合 → その内容を仕様書生成に反映する

該当ドキュメントが存在しない場合:

```
docs/features/{name}.md が見つかりません。
Mode B（対話形式）で情報収集し、完了後にドキュメント新規作成を提案します。
```

検出できなかった場合はエンジニアに確認する:

```
機能の種別を特定できませんでした。
該当するものを選んでください: 検索 / ページネーション / ループ / ナビゲーション / タブ / スライダー / カスタムブロック / その他
```

### Step 3: コンポーネント選択

`docs/components/index.md` を参照し、以下の優先順位でコンポーネントを決定する:

1. Figmaのフレーム名・コンポーネント名にBEMクラス（`c-xxx`）が明示 → そのクラスを採用
2. 明示がない場合 → `docs/components/index.md` を走査し、スタイルが一致するものがあれば採用
3. 同一スタイルが3箇所以上登場 → mixin化 or 変数化を仕様書に明記
4. 該当なし → 新規コンポーネントとして仕様書に追加

`docs/components/index.md` が存在しない場合:

```
docs/components/index.md が見つかりません（初回案件の場合は正常です）。
既存コンポーネントの確認をスキップして新規作成として進めます。
```

### Step 4: 仕様書・指示書・チェックリストを生成

`docs/specs/{feature-name}/` に以下の3ファイルを生成する:

**spec.md（仕様書）:**

```markdown
# {機能名} 仕様書

## 概要
{1文の説明}

## 対象ファイル
| ファイル | 種別 | 新規/修正 |
|---------|------|---------|

## コンポーネント一覧
| BEMクラス | 新規/既存 | バリアント |
|-----------|---------|----------|

## レスポンシブ仕様
| プロパティ | SP | PC |
|-----------|----|----|

## データ設計（ページ単位のみ）
### 投稿タイプ / カスタムフィールド
| フィールド名 | 型 | 必須 | 説明 |
|------------|----|----|-----|

### クエリ仕様
- ソート: {条件}
- 表示件数: {件}
- フィルタ: {条件}

## 未決事項
- {確認が必要な項目}
```

**implementation.md（実装指示書）:**

```markdown
# {機能名} 実装指示書

## 実装順序
1. {Step名}
...

## Step 1: {手順名}
**対象ファイル**: `{path}`
**変更内容**: {具体的な実装指示}

## コンポーネント登録
実装完了後、以下をdocs/components/index.mdに追記すること:
| BEMクラス | PHPパス | SCSSパス | 用途 |
|-----------|--------|---------|-----|

## 実装上の禁止事項
- raw pxでのfont-size・margin・padding指定
- raw breakpoint値の直書き（mixinを使うこと）
- デスクトップファースト記述（モバイルファースト必須）

## 自己検証チェック
- [ ] npm run lint がエラーゼロ
- [ ] SP表示確認
- [ ] PC表示確認
- [ ] PHP：esc_* エスケープが全出力に適用されているか
- [ ] PHP：function_exists() ラップがあるか
- [ ] img：loading="lazy" decoding="async" があるか
```

**checklist.md（完了チェックリスト）:**

```markdown
# {機能名} 完了チェックリスト

## 実装
- [ ] Step 1 完了
...

## 品質
- [ ] lint エラーゼロ（npm run lint）
- [ ] SP表示確認
- [ ] PC表示確認
- [ ] WordPress管理画面での操作確認

## ドキュメント
- [ ] docs/components/index.md に新規コンポーネントを追記
- [ ] tasks/lessons.md に気づきを記録
```

### Step 5: リファレンスドキュメントの作成・更新を提案

仕様書生成後、以下の条件でリファレンスドキュメントへの追記・新規作成を提案する。

**提案するタイミング:**

- Mode C で参照したドキュメントが空・未記入だった場合
- Mode B で収集した情報が今後の案件でも再利用できると判断した場合
- Mode A で Figma から新しい実装パターンを発見した場合

**提案フォーマット:**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📄 リファレンスドキュメントの更新提案
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
対象: docs/features/{name}.md

今回の実装から以下のコンテンツを追加することを提案します:

【実装パターン】
{今回採用した実装方針・アーキテクチャの概要}

【使用ライブラリ・プラグイン】
{今回使用したライブラリ・プラグインと選定理由}

【WordPress固有の注意点】
{wp_query・nonce・REST API等の注意事項で今回発見したもの}

【よくある実装ミス（予防）】
{今回の仕様書作成で気づいた落とし穴}

【チェックリスト追加項目】
- [ ] {今回固有の確認項目}

実装完了後、上記内容を docs/features/{name}.md に追記してください。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**ドキュメントが存在しない場合は新規作成を提案する:**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📄 リファレンスドキュメントの新規作成提案
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
docs/features/{name}.md が存在しません。
今回の実装をもとに以下の内容で新規作成することを提案します:

{テンプレートに今回の情報を埋め込んだ全文}

作成しますか？ [はい / いいえ]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

エンジニアが「はい」と答えた場合のみファイルを作成する。「いいえ」の場合は提案内容をチャットに残して終了する。

## エッジケース一覧

| 状況 | 対応 |
|------|------|
| Figma MCP 未接続（Mode A 指定時） | Mode B へフォールバック・エンジニアに通知 |
| `docs/components/index.md` が存在しない | スキップして新規作成として進める |
| `docs/features/{name}.md` が存在しない | Mode B へフォールバック → 完了後に新規作成を提案 |
| `docs/features/{name}.md` が空・未記入 | 空ドキュメントとして扱い → 完了後に追記内容を提案 |
| 機能キーワードが複数マッチ | 該当ドキュメントを全て参照し、エンジニアに確認 |
| 複数ページを一括で仕様書化したい | 1ページずつ実行するよう案内する |
| Figma MCPで取得できないデータ（フォントファミリー等） | 「未定」として記録し、エンジニアに確認を促す |

---

## トークン最適化

- 開始時にドキュメントを一括ロードしない
- `docs/features/{name}.md` は機能キーワード検出後、該当ファイルのみ参照
- `docs/components/index.md` はコンポーネント選択ステップ（Step 3）で初めて参照
- Figma データは必要なノードのみ取得する
