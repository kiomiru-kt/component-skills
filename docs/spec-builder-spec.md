# spec-builder スキル仕様書

**作成日**: 2026-04-09  
**ステータス**: 設計確定（スキル実装待ち）  
**関連スキル**: figma-extract, component-generate

---

## 概要

ページ・セクション・コンポーネント単位の実装仕様書と実装指示書を生成するスキル。  
Figma URLがあれば自動取得、なければ対話形式で情報を収集する。  
Claude Codeが仕様書を読んで即実装できる粒度の出力を目標とする。

---

## 課題背景

- 仕様書・要件定義が存在しない状態でエンジニアが実装着手している
- Figmaメモや口頭指示から仕様をエンジニアが手動で整理→工数2倍
- AIへのコンテキスト供給不足により独自実装・デザイン無視が発生
- フロントエンド実装（コンポーネント選択・レスポンシブ）に課題が集中

---

## 対応モード

### Mode A: Figma URLあり（自動取得）

```
エンジニアがFigma URLを渡す
    ↓
Figma MCPでデザインデータを自動取得
（ノード構造・色・サイズ・テキスト・スクリーンショット）
    ↓
不足情報のみ対話形式で補完
    ↓
仕様書・実装指示書を生成
```

### Mode B: Figma URLなし（手動入力）

```
対話形式でFAQ形式の質問を順番に実施
（1問ずつ。一度に複数質問しない）
    ↓
収集した情報を整理・確認
    ↓
仕様書・実装指示書を生成
```

---

## 対応単位

### ページ・セクション単位
対象：トップページ、一覧ページ、詳細ページ、固定ページ等  
出力：ページ全体の構成・テンプレート・使用コンポーネント一覧を含む仕様書

### コンポーネント単位
対象：c-card、c-btn、c-nav等の個別コンポーネント  
出力：PHP template-part + SCSSのペア実装に必要な仕様書

### 機能単位
対象：検索・ページネーション・スライダー等の機能実装  
出力：機能リファレンスドキュメントを参照した精度の高い実装仕様書

### 参照タイミング

- **自動検出**：spec-builderが機能名・キーワードを検出したら該当ドキュメントを自動参照
- **明示指定**：エンジニアが「search機能で」と指定した場合も参照

両方対応。検出できなかった場合は確認を挟む。

### 運用方針

- **初期状態**：テンプレートのみ（実装例なし）
- **プロジェクト毎に育てる**：実装完了後にエンジニアが知見を追記
- **テンプレートはボイラープレートにコミット**：`kiomiru-template-wordpress/docs/features/`
- **プロジェクト固有の知見**：各案件の `docs/features/` に直接追記して育てる

## 情報収集項目（FAQ）

以下を**1問ずつ**順番に確認する。不明な項目は「未定」として記録し進める。

### 共通（ページ・コンポーネント両方）

1. **対象の概要**: 何を作るのか（1文で）
2. **Figma URL**: あれば自動取得。なければ次へ
3. **レスポンシブ仕様**:
   - SP / PC のデザイン差分
   - ブレークポイント（プロジェクトのmixin使用前提）
4. **インタラクション**: hover・active・アニメーション有無

### ページ・セクション固有

5. **WordPressテンプレート**: archive / single / page / front-page / template-parts
6. **使用コンポーネント**: docs/components/index.mdを参照し候補を提示
7. **データ取得**:
   - 投稿タイプ（既存 or CPT）
   - ACF/SCFカスタムフィールド一覧
   - WP_Queryパラメータ（件数・ソート・フィルタ）
   - ページネーション有無
8. **SEO**: メタ情報・構造化データの有無

### コンポーネント固有

5. **BEMクラス名**: docs/components/index.mdで既存確認 → 新規 or 既存
6. **バリアント**: suffix規則に基づく状態一覧（default/active/PC/SP等）
7. **引数（$args）**: PHPテンプレートに渡すデータ項目

## コンポーネント選択ルール（必須チェック）

仕様書生成前に以下の順序でコンポーネントを決定する：

```
1. Figmaのフレーム名・コンポーネント名にBEMクラス（c-xxx）が明示
   → そのクラスを採用

2. 明示がない場合
   → docs/components/index.mdを走査
   → スタイルが一致するコンポーネントがあれば採用

3. 同一スタイルが3箇所以上登場
   → mixin化 or 変数化を仕様書に明記

4. 該当なし
   → 新規コンポーネントとして仕様書に追加
   → 実装後にdocs/components/index.mdへの追記を指示書に含める
```

## レスポンシブ仕様の明示化ルール

仕様書には必ず以下の形式でレスポンシブ差分を記載する：

```markdown
### レスポンシブ仕様
| プロパティ | SP | PC |
|-----------|----|----|
| font-size | fluid-font-rem(14, 16) | fluid-font-rem(16, 18) |
| padding | 12px 16px | 20px 24px |
| gap | 16px | 24px |
| columns | 1列 | 3列 |
```

**禁止事項**（実装指示書に明記）：
- raw pxでのfont-size・margin・padding指定
- raw breakpoint値の直書き（mixinを使うこと）
- デスクトップファースト記述（モバイルファースト必須）


### spec.md テンプレート

```markdown
# {機能名} 仕様書

## 概要
{1文の説明}

## 対象ファイル
| ファイル | 種別 | 新規/修正 |
|---------|------|---------|

## コンポーネント一覧
| BEMクラス | 新規/既存 | バリアント |
|-----------|---------|----------|

## レスポンシブ仕様
| プロパティ | SP | PC |
|-----------|----|----|

## データ設計（ページ単位のみ）
### 投稿タイプ / カスタムフィールド
| フィールド名 | 型 | 必須 | 説明 |
|------------|----|----|-----|

### クエリ仕様
- ソート: {条件}
- 表示件数: {件}
- フィルタ: {条件}

## 未決事項
- {確認が必要な項目}
```

### implementation.md テンプレート

```markdown
# {機能名} 実装指示書

## 実装順序
1. {Step名}
2. {Step名}
...

## Step 1: {手順名}
**対象ファイル**: `{path}`  
**変更内容**: {具体的な実装指示}

## コンポーネント登録
実装完了後、以下をdocs/components/index.mdに追記すること：
| BEMクラス | PHPパス | SCSSパス | 用途 |
|-----------|--------|---------|-----|

## 自己検証チェック
- [ ] npm run lint がエラーゼロ
- [ ] SP表示確認
- [ ] PC表示確認
- [ ] レスポンシブ：pxの直書きがないか
- [ ] PHP：esc_* エスケープが全出力に適用されているか
- [ ] PHP：function_exists() ラップがあるか
- [ ] img：loading="lazy" decoding="async" があるか
```

### checklist.md テンプレート

```markdown
# {機能名} 完了チェックリスト

## 実装
- [ ] Step 1 完了
- [ ] Step 2 完了
...

## 品質
- [ ] lint エラーゼロ（npm run lint）
- [ ] SP表示確認
- [ ] PC表示確認
- [ ] WordPress管理画面での操作確認

## ドキュメント
- [ ] docs/components/index.md に新規コンポーネントを追記
- [ ] tasks/lessons.md に気づきを記録
```

## Figma MCPの使い方（Mode A）

```
1. get_metadata でノード構造を取得
2. get_design_context で対象フレームの詳細を取得
3. get_screenshot でビジュアルを確認
4. 取得できなかった情報のみ対話で補完
```

取得した数値は仕様書のレスポンシブテーブルにそのまま記載する。  
AIによる推測・補完は禁止。取得できない場合は「未定」として記録。

## 制約

- WordPress Coding Standardsに準拠した実装を指示すること
- docs/theme-architecture.mdのファイル責務ルールに従うこと
- docs/scss-guidelines.mdのFLOCSS+BEM規約に従うこと
- 新規コンポーネントは必ずPHP + SCSSのペアで指示すること
- 実装指示書には必ずレスポンシブ仕様テーブルを含めること
